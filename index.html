Track 1 · The Origins

Cloud security started with one idea: block the packet at the door

When workloads first moved to the cloud, the security model came straight from the network layer — Security Groups and Network Security Groups (NSGs). The founding philosophy: if you control which packets get in, you've done your job.

Security Groups / NSGs

A 5-tuple allow/deny list

Source, destination, port, protocol, direction. Stateful filtering at the edge of every subnet and NIC — fast, cheap, and built into the fabric.

The founding philosophy

"Blocking traffic is good enough"

If the wrong packets never arrive, the thinking went, the workload is safe. Connectivity control was treated as a complete security model.

Blocking was only ever half the job

A 5-tuple rule decides whether a packet gets in. It says nothing about what's inside the sessions you deliberately allow.

Expose port 443 to deliver a web app, and an NSG gives you exactly zero inspection of what rides inside that permitted tunnel. The traffic you let in is the blind spot.

Track 3 · The Framework

OWASP Top 10 for LLMs

Before we walk through the risks, the framework we'll measure them against — the shared language security teams, vendors, and auditors have standardized on.

WHAT IT IS

A community-built, vendor-neutral catalogue of the ten most critical security risks in LLM and agent applications — maintained by OWASP, the same body behind the web AppSec Top 10.

WHY IT EXISTS

Traditional AppSec frameworks never accounted for prompts, model outputs, or autonomous tool use. Teams adopting AI had no shared way to name or compare these risks. This closed that gap.

WHY IT'S THE BENCHMARK

It's vendor-neutral and CISO-recognized — now the common reference in RFPs, audits, and architecture reviews. If you secure AI, this is the language you're expected to speak.

LLM01

Prompt Injection

Adversarial input overrides instructions, incl. indirect injection via documents & data.

LLM02

Insecure Output Handling

Downstream systems trust unvalidated model output.

LLM03

Training Data Poisoning

Tampered data corrupts model behavior.

LLM04

Model Denial of Service

Token/compute abuse drives cost and outage.

LLM05

Supply Chain

Risky third-party models, plugins, and datasets.

LLM06

Sensitive Information Disclosure

Secrets, PII, and source code leave in prompts.

LLM07

Insecure Plugin Design

Unsafe tool/plugin interfaces the agent can invoke.

LLM08

Excessive Agency

The agent can take actions it should never have been able to.

LLM09

Overreliance

Humans trust confident-but-wrong AI output.

LLM10

Model Theft

Unauthorized extraction of proprietary models.

Maintained by OWASP — the catalogue we'll map each risk back to as we move through the talk.

Section 6 · From Data Plane to Control Plane

"Secure or not" is often just a toggle

  • The on-prem mental model = a clearly defined edge; we try to replicate it in cloud
  • But cloud is self-service and identity / IAM-driven — much lives in the control plane
  • Whether something is secure often = which position a single toggle is in

You can't insert a firewall in front of a misconfiguration.

Cloud storage (S3, GCS, Azure Blob) is among the most commonly used — and most commonly misconfigured — cloud resources. Data flows in and out over the internet; it never passes through the firewall, so the NGFW has zero visibility into whether a bucket is public or private.

A different problem that needs a different tool — starting with visibility into how things should be done.
